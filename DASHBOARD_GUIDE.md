# 中转站监控系统：登录、分组与新增站点经验指南

本指南详细总结了本监控大屏系统对各大中转站进行静默余额抓取、真实分组倍率提取的实现原理与使用方法，供未来维护和扩展时参考。

---

## 一、 网页免登与数据抓取原理

### 1. 为什么以前部分站点抓取失败？
* **OIDC 误登出干扰**：部分站点带有 "OIDC" 或 "第三方授权" 字样，旧脚本一律进行点击，反而导致已有登录态的 Session 被强行注销。
* **特殊官方站/免费接口**：如英伟达（nvidia.com）免费接口没有余额账单，需要作为免费站短路放行，不再做无意义的网页抓取。
* **新中转系统的 Token 差异**：以 `tygzs` 和 `riyuexy`（日月付费）为代表的新版中转系统（Sub2API），其认证 JWT 是存储在 `localStorage.getItem('auth_token')` 中，而非 One-API 标准的 `token`，导致先前发出 API 请求时请求头缺失 `Authorization` 字段而被拦截。

### 2. 共享 Chrome 登录态（免登）
本系统依赖一个处于调试模式（CDP 开启，监听 `9222` 端口）的 Chrome 实例：
* **数据目录共享**：通过命令启动的 Chrome 使用了 `A:\ChromeDevToolsProfile` 作为用户数据目录（Profile）。
* **登录态保存**：一旦您在 Chrome 中登录了任何中转站账号（包括复杂的 OIDC 重定向认证），该状态将永久保存在此 Profile 目录下。
* **静默读取**：后台 Python 使用 Playwright 连接到该 Chrome 实例，在无头或静默状态下同源 fetch 其 API。

## 二、 真实分组与倍率提取机制

每个中转站在大屏卡片上以“子项”列表展示该站点的真实分组名称、延迟和价格项，不再使用 ccswitch 中的 provider 记录充当假分组。

### 1. 提取规则
* **支持多分组的 API 站点 (One-API / New-API)**：
  - 同源请求 `/api/pricing` 或 `/api/token/prices` 获取各分组的名称及其倍率（如 `default (1.0x)`，`vip (2.0x)`）。
  - 若以上接口被禁，使用 `/api/user/self` 或 `/api/user/info` 提取当前用户所属的 `group` 字段作为真实分组。
* **无多分组的站点 (新中转系统 Sub2API)**：
  - 新中转系统通过 `/api/v1/auth/me` 接口获取当前余额。因为不支持多分组机制，系统会读取其在 ccswitch 中填写的 `cost_multiplier`（倍率），并默认命名为 `default` 分组展现（例如 `default (0.09x)`）。
* **展示无限制**：
  - 彻底取消大屏上仅展示前 4 个最低倍率分组的限制，实现“有几个真实分组，大屏就全部渲染几个”。

## 三、 未来如何新增中转站？

如果以后希望将新站点加入大屏，只需按以下步骤操作：

### 步骤 1：在 ccswitch 中导入/新增站点
在您的 CCSwitch 客户端中导入或创建该站点提供商，配置其真实的 `Anthropic Base URL` 及 `API Key`，并填写正确的 `倍率/乘数`（`cost_multiplier`）和 `官网地址`（`website_url`）。系统在启动时会自动读取 CCSwitch 的 SQLite 数据库并显示新卡片。

### 步骤 2：对新站点进行一次手动登录
1. 运行工作目录下的 `Start-Claude-Chrome.cmd` 脚本启动调试版 Chrome 浏览器。
2. 在该浏览器中，打开新站点的官网并完成登录。
3. 确认您可以正常查看用户的额度/个人资料页面，然后关闭或最小化 Chrome（请勿点击“退出登录”）。

### 步骤 3：智能识别与大屏更新
新站点的登录态（Cookie、LocalStorage 和 Session）会被自动持久化。当您在大屏点击“手动刷新”时，Playwright 会自动以同源 fetch 读取新站的真实余额和分组，并全量展示在大屏上，全程无需编写任何新代码。
